# MoSCoW Prioritization (ProductPlan)

The MoSCoW method is a prioritization framework used to help key stakeholders understand the significance of initiatives in a release.

Created by software development expert **Dai Clegg** while working at Oracle. Originally documented in the Dynamic System Development Method (DSDM) handbook.

## Categories

### Must-have
Non-negotiable needs. If the product won't work without an initiative, it's a must-have. Example: security functionalities for a healthcare app.

### Should-have
Essential but not vital. If left out, the product still functions. Example: performance improvements, minor bug fixes.

### Could-have
Nice-to-haves. Smallest impact if left out. First to be deprioritized if higher-category items expand.

### Will not have (this time)
Manages expectations about what's excluded. Helps prevent scope creep. Some items get prioritized in future releases; others don't.

## Applications Beyond Time Boxing

- **Budgetary constraints**: Use must-haves and should-haves to determine what fits within budget
- **Team skillsets**: Constrain prioritization by developer expertise
- **Competing company needs**: Use MoSCoW when multiple executive priorities compete

## Drawbacks

1. **Inconsistent scoring** — MoSCoW doesn't include an objective methodology for ranking. Needs pairing with weighted scoring, value vs. complexity, Kano model, etc.
2. **Missing stakeholders** — Excluding relevant stakeholders leads to miscategorized items
3. **Team bias** — Without objective scoring, personal opinions can undermine effectiveness

## Best Practices

1. Choose an objective ranking system (weighted scoring, Kano model, RICE, ICE)
2. Seek input from all key stakeholders
3. Share the MoSCoW process across the organization for consensus
